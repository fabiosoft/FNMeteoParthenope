Getting Started with FNMeteoParthenope

This is a guide to help developers get up to speed with FNMeteoParthenope. It is geared primarily towards anyone who is new to Mac or iOS development, or has not worked extensively with 3rd-party libraries before.

These step-by-step instructions are written for Xcode 4.6+, using either the iOS 6 or Mac OS 10.7 SDK. If you are using a previous version of Xcode, you may want to update before starting.

Download FNMeteoParthenope

Step 1: Download FNMeteoParthenope

First, we need to download the latest source code from the FNMeteoParthenope repository.

Click here to download FNMeteoParthenope in a zip archive**

Once the download has finished, unarchive the zip file and open the resulting directory. The folder we'll be interested in later is named "FNMeteoParthenope", but hold off on doing anything with that for a moment.

Add FNMeteoParthenope

Step 2: Add FNMeteoParthenope to Your Project

Once we have the latest version of FNMeteoParthenope, we're ready to add it to our project.

Now, on a side of screen go to that unarchived folder from the code we downloaded a few minutes ago, select and drag FNMeteoParthenope.framework and Networking.framework into Framework folder.

Check that the options below the file browser are as follows:

  • Destination ☑ Copy items into destination group's folder (if needed) [Checked]
  • Folders ◉ Create Groups for any added folders [Top Item Selected]
  • Add to Targets ☑ (Project Name) [All Targets Selected]

Finally, a new "FNMeteoParthenope" framework will have been added to your project, which corresponds to the folder that is now in your project directory. You can also choose to reference to the folder, but don't forget to bring all with you if you backup.

Now, a very important step: we need to add the _-ObjC_ to Other Linker Flag. If you don't know how to do that follows this image.

It's not a my choose, Apple say to do that when you are some categories into the project. See it here a detailed explaination by Apple.

Check That Everything Is Working

Step 3: Check That Everything Is Working

Click on the Targets → Your app name → and then the 'Build Phases' tab and then expand 'Link Binary With Libraries' as shown.

Add the following libraries:

  • CoreLocation.framework
  • MobileCoreServices.framework (or CoreServices.framework if your target is Mac OS)
  • SystemConfiguration.framework

Now that our project is all set up with FNMeteoParthenope, let's try building the project. Select "Project > Build", or use its keyboard shortcut, ⌘B.

Let's go!

Step 4: Let's go!

At this point, everything's in place for you to start using FNMeteoParthenope. Just #import the headers for the classes you need and get to it! Move to -Prefix.pch file and add these lines inside #ifdef __OBJC__ statement

#import <SystemConfiguration/SystemConfiguration.h>
#import <MobileCoreServices/MobileCoreServices.h>
#import <FNMeteoParthenope/FNMeteoClient.h>

Don't know where to start, exactly? Well, that really depends on what your app does. Listed below are a few common tasks, with an explanation of how to use FNMeteoParthenope to accomplish them.

Example

Get weather forecast for now

getWeatherForecastForNowPlace is the perfect method/tool for this job. All you need to do is ask it for a particular FNPlace, and it'll give you an FNFeedBase object containig all entries.

FNPlace *thePlace = [[FNPlace alloc]initWithIdentifier:@"prov063" label:@"Napoli"];
[[FNMeteoClient sharedInstance]getWeatherForecastForNowPlace:thePlace model:HIGH_RES_WEATHER_24H_WRF5 output:GENERAL success:^(FNMeteoRequestOperation *operation, FNFeedBase *response) {
NSLog(@"%@",response.entries[0]);
} failure:^(FNMeteoRequestOperation *operation, NSError *error) {
NSLog(@"%@",error.localizedDescription);
}];

First, we construct an FNPlace object. It encapsulates pretty much everything you need for make a server request.

We then pass that object to place object into FNMeteoClient sharedInstance-getWeatherForecastForNowPlace:success:failure.

The success and failure parameters take block arguments. Blocks are a relatively new and complex feature in Objective-C, so if you're unfamiliar with them, think about reading up on them. Blocks are objects that contain a specified behavior–code that can be executed later.

In the case of our success and failure callback blocks, we're specifying what should happen after the request finishes. The request happens in the background, without locking up or slowing down the app. Depending on whether the response from the server indicates success (e.g. 200 status code and JSON Content-Type), or failure (e.g. 404 status code or unknown MIME type), the code in the corresponding block will be executed.
